# Peer Review: Meal Planner Feature

**Date:** 2026-05-09
**Reviewer:** Gemini CLI (Peer Review Role)

## 1. Acceptance Criteria Evaluation

| Story / Requirement                | Status     | Observations                                                                                         |
| :--------------------------------- | :--------- | :--------------------------------------------------------------------------------------------------- |
| **Story 0: Tenancy & Local Time**  | ✅ Pass    | `userId` checks are consistent. Local time handling is present but could be more robust.             |
| **Story 1: Calendar & Scheduling** | ⚠️ Partial | Basic scheduling works. **AC 3 (Clickable recipes with scale)** is missing; titles are static spans. |

## 2. Bug Fixes Verification

Based on the review of the modified files, the following bugs are verified as **Resolved**:

- **BUG-014 to BUG-017**: All UI/UX adjustments for search, circular reporting, reordering, and alignment are implemented.
- **BUG-018 to BUG-021**: Timer persistence, dismissal, and "Exit Mode" logic in `RecipePlayMode` are functional.
- **BUG-022 to BUG-024**: Dashboard grid, responsiveness, and deletion logic are implemented.
- **BUG-025 to BUG-026**: Recipe validation and step filtering are enforced.

## 3. Code Quality & Structure

- **Database Schema**: The Prisma schema is well-architected. The inclusion of `PrepCompletion` and `sourcePlannedRecipeId` shows strong alignment with the long-term plan.
- **Library Logic (`lib/meal-plans.ts`)**: Core functions are clean but incomplete. `getPrepAheadData` lacks the recursive depth and unit normalization logic necessary for professional kitchen accuracy.
- **UI Components**: `MealCalendarClient` is growing too large (~200 lines). It should be split into smaller, focused components (e.g., `MealSlot`, `PlannedRecipeRow`).

## 4. Edge Cases & Missed Items

- **E2E Testing**: **CRITICAL OMISSION.** Despite the feature plan's "Mandatory" requirement, no E2E tests exist in `tests/e2e/` for the Meal Planner.
- **Circular Dependencies**: No logic exists to prevent a recipe from being its own leftover source (directly or indirectly).
- **Date Integrity**: Users can plan to eat leftovers before the source meal is cooked.
- **Unit Normalization**: The aggregator sums quantities without converting units (e.g., adding 1kg and 1lb results in "2" of an undefined unit).
- **Scaling Context**: Navigation from the calendar to the recipe view does not yet pass the `scale` parameter, failing the user journey described in Story 1.

## Recommendations

1.  **Implement E2E Suite**: Add `tests/e2e/meal-planner.spec.ts` immediately to cover tenancy and scheduling.
2.  **Decompose Calendar UI**: Refactor `MealCalendarClient` into sub-components for maintainability.
3.  **Harden Aggregator**: Update `getPrepAheadData` to normalize units using `lib/units.ts` and recursively resolve sub-recipes.
4.  **Add Validation**: Implement date and circularity checks in the `linkLeftoverConsumptionAction`.
5.  **Complete Missing ACs**: Prioritize Meal Cloning (Story 5) and Macro Aggregation (Story 6) to finish the feature set.
