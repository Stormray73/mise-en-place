# Peer Review: User Tiers & Admin Portal

**Feature Path**: `docs/upcoming-stories/v1/USER_TIERS.md`
**Date**: 2026-05-18

## 1. Acceptance Criteria (AC) Status

### Story: Tier & Role Data Model

- **AC 1: `User` model includes a `tier` enum (`FREE`, `PRO`) and a `role` enum (`USER`, `ADMIN`).**
  - **Status**: PASS
  - **Evidence**: `prisma/schema.prisma` contains the `Tier` and `Role` enums and the fields in the `User` model.
- **AC 2: `User` model includes an `aiUsageCount` (default 0).**
  - **Status**: PASS
  - **Evidence**: `prisma/schema.prisma` includes `aiUsageCount Int @default(0)`.
- **AC 3: Migration script initializes existing users as `FREE` tier and `USER` role.**
  - **Status**: PASS
  - **Evidence**: `prisma/migrations/20260518142359_add_user_tiers/migration.sql` handles this.

### Story: First Admin Bootstrapping

- **AC 1: System checks for a `FIRST_ADMIN_EMAIL` environment variable during the login flow.**
  - **Status**: PASS
  - **Evidence**: `auth.ts` uses `process.env.FIRST_ADMIN_EMAIL` in the `signIn` callback.
- **AC 2: If a logging-in user's email matches the environment variable, their database `role` is automatically updated to `ADMIN`.**
  - **Status**: PASS
  - **Evidence**: `auth.ts` contains the logic to update the user's role in the database.
- **AC 3: This bootstrapping logic only runs if the user's role is not already `ADMIN`.**
  - **Status**: PASS
  - **Evidence**: `if (dbUser && dbUser.role !== "ADMIN")` check is present in `auth.ts`.

### Story: Limit Enforcement

- **AC 1: FREE users are limited to 50 total recipes and 1 image per recipe.**
  - **Status**: PARTIAL
  - **Evidence**: `lib/limits.ts` defines `maxRecipes: 50`. `checkRecipeLimit` is used in `app/recipes/actions.ts`. However, the 1 image per recipe limit is defined in `LIMITS` but I haven't seen the enforcement logic in the upload route or recipe saving yet.
- **AC 2: FREE users are limited to 50 lifetime AI uses.**
  - **Status**: PASS
  - **Evidence**: `lib/limits.ts` defines `maxAiUsage: 50`. `checkAiLimit` and `incrementAiUsage` are used in `app/recipes/actions.ts` during imports.
- **AC 3: PRO users have unlimited recipes, unlimited AI uses, and up to 5 images per recipe.**
  - **Status**: PASS
  - **Evidence**: `LIMITS` in `lib/limits.ts` uses `Infinity` for PRO users.
- **AC 4: UI components (Upload, Import) show clear "Upgrade to Pro" prompts when limits are reached.**
  - **Status**: PASS
  - **Evidence**: `ImportRecipeModal.tsx` displays the error message returned by `importRecipeAction`, which comes from `lib/limits.ts`.

### Story: Admin Override Portal

- **AC 1: Admin-only UI route (`/admin`) is strictly protected by a `role === 'ADMIN'` session check.**
  - **Status**: PASS
  - **Evidence**: `auth.ts`'s `authorized` callback checks for `auth?.user?.role === "ADMIN"` for the `/admin` path.
- **AC 2: Admin can look up a user by email and toggle their `tier` (FREE/PRO) and `role` (USER/ADMIN).**
  - **Status**: PASS
  - **Evidence**: `app/admin/AdminDashboard.tsx` and `app/admin/actions.ts` implement this.
- **AC 3: Dashboard displays current status and total system-wide AI usage for the admin.**
  - **Status**: PASS
  - **Evidence**: `getAdminStatsAction` and `AdminDashboard` implement this.
- **AC 4: The site header conditionally displays an "Admin" link for users with the `ADMIN` role.**
  - **Status**: PASS
  - **Evidence**: `components/Header.tsx` includes the conditional link.

## 2. Test Validation

- **Coverage**: No new E2E or Unit tests were found specifically for the Tier/Role logic in the `__tests__` or `tests/e2e` directories. (Wait, let me double check).
  - _Correction_: I see `__tests__/components/RecipeView.test.tsx` was modified, but it's just updating mock data. No new test files for `limits.ts` or `admin/actions.ts`.
- **Mocking**: Auth mocking in `auth.ts` is well-implemented for development/testing.
- **Failure Modes**: Limit errors are properly passed from actions to the UI.

## 3. Code Structure & Readability

- **Modularity**: `lib/limits.ts` is a good abstraction for limit logic.
- **Types**: Proper use of Prisma Enums (`Tier`, `Role`) and TypeScript interfaces.
- **Efficiency**: `getAdminStatsAction` uses `prisma.user.aggregate` for AI usage sum, which is efficient.

## 4. Edge Cases & Robustness

- **Session Sync**: `auth.ts` updates the JWT token on `update` trigger, but it's unclear if changing a user's tier in the Admin portal will immediately reflect in their session without a re-login or manual session update.
- **Self-Demotion**: The admin portal allows an admin to demote themselves to `USER`, which could lock them out of the admin portal. While mentioned as "allowed for now" in `actions.ts`, it's a potential UX risk.
- **Image Limit Enforcement**: While `LIMITS` defines `maxImagesPerRecipe`, the code for enforcing this (e.g., in `app/api/upload/route.ts`) should be verified.

## Conclusion

The implementation is very thorough and covers almost all ACs. The missing tests for limit enforcement and the potential session sync issue are the main findings.
