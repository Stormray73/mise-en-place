## 2. Code Quality & Standards

- **Idiomatic Patterns**: The implementation follows the established Next.js App Router and Server Actions patterns. The use of a shared modal (`CustomIngredientModal`) across search and dashboard is excellent.
- **Type Safety**: The `USDAFood` interface was effectively extended to handle both external and internal data sources.
- **Efficiency**: Database queries are targeted. The search route uses `insensitive` contains, which is appropriate for this scale.
- **UX**: UI remains consistent with the "Kitchen UI" aesthetic. `confirm()` for deletion is a simple but effective safety measure.

## 3. Edge Cases & Robustness

### Identified Issues/Gaps:

1. **Server-Side Validation**: `saveCustomIngredientAction` lacks server-side validation for macro values. While the client-side UI prevents negative numbers, a malicious or malformed request could bypass this.
2. **Duplicate Ingredient Names**: There is no check to prevent a user from creating multiple custom ingredients with the exact same name, which could lead to confusion in the search results.
3. **Unit Conversion Fragility**: The system stores the custom unit as the primary `gramWeight` in `foodPortions`. While this works for ratio-based macro calculation, it may cause issues if the system later implements strict unit conversion between grams and "pieces" for custom items.
4. **Search Scalability**: The custom ingredient search is limited to `take: 10`. While fine for now, users with many custom ingredients might not find what they need if the query is too broad.

### Test Coverage Gaps:

- **Unit Testing**: `__tests__/api/usda/search.test.ts` was not updated to verify the merging logic.
- **E2E Testing**:
  - No test for editing an existing custom ingredient.
  - No test for the prevention of deleting an ingredient used in a recipe.

## 4. Recommendations

- **Security**: Add server-side Zod validation for `CustomIngredientSaveData` to ensure `baseAmount > 0` and `macros >= 0`.
- **UX**: Add a unique constraint check or a warning if a user tries to create an ingredient with a name they've already used.
- **Testing**: Expand `custom-ingredients.spec.ts` to cover the "Delete Prevention" logic by attempting to delete an ingredient used in a "Test Recipe".
- **Testing**: Update unit tests for the USDA search API to mock Prisma and verify custom ingredients are returned.
